ตัวเลือก ER Diagram 6 แบบ

เลือก layout ที่อ่านง่ายก่อนนำไปแทน ER ในหน้า Data Dictionary จริง ทุกแบบตั้งใจให้เส้นไม่ลากทับกล่องตาราง

กลับ Data Dictionary DB และ Workflow User Flow Diagram
ตารางที่ใช้ในทุกแบบ

bod_cod_deviation_reports เป็นตารางหลัก และเชื่อมกับ measurements, reviews, attachments, approval_steps, approval_events โดย approval_events เชื่อมกับ approval_steps เพิ่มอีกเส้นหนึ่ง

แบบที่ 1: Hub and Spoke

วาง reports เป็นศูนย์กลาง เส้นสั้นและวิ่งออกขอบแต่ละด้าน อ่านความเป็นตารางหลักได้เร็ว

แนะนำสำหรับเอกสารสรุป
จุดเด่น: เส้นไม่ผ่านกลางตารางเหมาะกับรายงานผู้บริหาร

แบบที่ 2: Left to Right Pipeline

อ่านจากซ้ายไปขวา: รายงาน → ข้อมูลประกอบ → การอนุมัติ/ประวัติ เหมาะกับการอธิบายลำดับข้อมูล

แนะนำสำหรับ flow เชิงระบบ
จุดเด่น: เส้นเป็นแนวนอน/ฉากอ่านง่ายบนจอกว้าง

แบบที่ 3: Top Down Bus

ใช้เส้น bus กลางจาก reports ลงมาแตกแขนง ทำให้ไม่มีเส้นเฉียงตัดกล่อง

แนะนำถ้าต้องการความเป็นระเบียบ
จุดเด่น: ไม่มีเส้นเฉียงเหมาะกับการพิมพ์ A4 แนวนอน

แบบที่ 4: Grouped by Responsibility

แบ่งกลุ่มข้อมูลรายงาน, เอกสาร, และ approval/audit ชัดเจน เหมาะกับคุยกับทีม backend

แนะนำสำหรับออกแบบ DB จริง
จุดเด่น: แยก responsibility ชัดเหมาะกับอธิบาย module

แบบที่ 5: Compact Table Cards

ลด field ในกล่องให้สั้น ใช้ relationship summary ด้านล่างช่วย รายงานจะไม่แน่นและเส้นไม่ชนข้อความ

แนะนำถ้าหน้ารายงานต้องไม่รก
จุดเด่น: โล่งที่สุดเหมาะกับ slide/เอกสารประกอบ

แบบที่ 6: Report-Centric Ring

วาง child รอบ reports แบบสมดุล ใช้เส้นฉากออกจากขอบ ไม่ตัดกลางกล่อง

แนะนำถ้าต้องการภาพสวย
จุดเด่น: สมดุลและดูเป็นภาพรวมเส้นไม่ทับกล่อง
