ออกแบบระบบข้อมูลการแจ้งเตือน CEMS และ BOD/COD Online

เอกสารนี้สรุปแนวทางเก็บข้อมูล, flow งาน, schema, API สำหรับทีมภายนอก POST ค่าเข้ามา, และแนวทางรองรับข้อมูลขนาดใหญ่ระดับหลายหมื่นโรงงานต่อวัน

Source หลัก: 60m raw data ผลลัพธ์กลาง: alert_events กันซ้ำ: idempotency_key Scale: batch + summary

1. ข้อสรุปการออกแบบ

แนะนำให้ใช้ ตารางกลางตารางเดียวสำหรับเหตุการณ์แจ้งเตือน คือ alert_events แล้วแยกหน้าจอด้วย alert_type, system_type, view หรือ API filter แทนการสร้างตารางแยกตาม alert ทุกประเภท

ไม่ควรให้หน้าเว็บหรือ API รายงานไป scan ตาราง {stationId}_data_60m สด ๆ ทุกครั้ง เพราะข้อมูลจะใหญ่ตามจำนวนโรงงาน จุดตรวจ และพารามิเตอร์ ควรให้ background job คำนวณไว้ล่วงหน้า แล้วหน้าเว็บอ่านจาก alert_events
Raw data เก็บค่าตรวจวัดจำนวนมากใน Parameter DB เช่น ingest.S0001_data_60m
Daily summary สรุป completeness, no report, abnormal ต่อวัน เพื่อให้ job อ่านเร็ว
Alert event เก็บเฉพาะเหตุการณ์ที่เข้าเงื่อนไขแจ้งเตือน พร้อม snapshot สำหรับเปิดย้อนหลัง

2. ประเภทการแจ้งเตือน

กลุ่ม รายการแจ้งเตือน alert_type แหล่งข้อมูลที่แนะนำ
CEMS ผลการตรวจวัด CEMS เกินมาตรฐาน STANDARD_EXCEEDED รับ POST จาก API ภายนอกเท่านั้น
CEMS ผลการตรวจวัด CEMS เกินค่า EIA กำหนด EIA_EXCEEDED รับ POST จาก API ภายนอกเท่านั้น
CEMS รายงานผล CEMS ไม่ถึงร้อยละ 80 ต่อวัน DAILY_COMPLETENESS_LOW job 08:30 คำนวณจากข้อมูลวันก่อนหน้า ด้วยสูตร Normal / (24 - Shut Down)
CEMS ไม่รายงานผล CEMS ติดต่อกันเกิน 14 วัน CONSECUTIVE_NO_REPORT เริ่มแจ้งวันที่ 15 และแจ้งทุกวันจนกว่าจะมีค่ากลับเข้ามา
CEMS รายงานผลตรวจวัด CEMS มีค่าผิดปกติ ABNORMAL_VALUE rolling window จาก 60m: ค่านิ่ง 5 ครั้งติดกัน, ค่า 0 5 ครั้งติดกัน, ค่าติดลบ 5 ครั้งติดกัน
BOD/COD Online ผลการตรวจวัด BOD/COD Online เกินมาตรฐาน STANDARD_EXCEEDED รับ POST จาก API ภายนอกเท่านั้น
BOD/COD Online ผลการตรวจวัด BOD/COD Online เกินค่า EIA กำหนด EIA_EXCEEDED รับ POST จาก API ภายนอกเท่านั้น
BOD/COD Online รายงานผล BOD/COD Online ไม่ถึงร้อยละ 80 ต่อวัน DAILY_COMPLETENESS_LOW job 08:30 คำนวณจากข้อมูลวันก่อนหน้า ด้วยสูตร Normal / (24 - Shut Down)
BOD/COD Online ไม่รายงานผล BOD/COD Online ติดต่อกันเกิน 7 วัน CONSECUTIVE_NO_REPORT เริ่มแจ้งวันที่ 8 และแจ้งทุกวันจนกว่าจะมีค่ากลับเข้ามา
BOD/COD Online รายงานผลตรวจวัด BOD/COD Online มีค่าผิดปกติ ABNORMAL_VALUE rolling window จาก 60m: ค่านิ่ง 5 ครั้งติดกัน, ค่า 0 5 ครั้งติดกัน, ค่าติดลบ 5 ครั้งติดกัน
กติกา no report: ถ้า CEMS ไม่มีค่ารายงานติดต่อกันเกิน 14 วัน ให้เริ่มสร้าง alert ในวันที่ 15 และสร้างต่อทุกวันจนกว่าจะมีค่ากลับเข้ามา ถ้า BOD/COD Online ไม่มีค่ารายงานติดต่อกันเกิน 7 วัน ให้เริ่มสร้าง alert ในวันที่ 8 และสร้างต่อทุกวันจนกว่าจะมีค่ากลับเข้ามา เมื่อมีค่ากลับเข้ามาอย่างน้อย 1 ค่า ให้ reset การนับ no report ใหม่ ถ้าหลังจากนั้นค่าหายไปอีก หรือค่าที่เข้ามากลายเป็นค่านิ่ง/ค่า 0/ค่าติดลบ ให้เริ่มนับ sequence ใหม่จากหลังจุด reset นั้น
กติกา 80%: คำนวณจากจำนวนค่า status Normal เทียบกับจำนวนชั่วโมงที่ควรมีข้อมูล หลังหักช่วง Shut Down ออก สูตรคือ normalStatusCount / (24 - shutDownStatusCount) * 100 ถ้าผลลัพธ์ต่ำกว่า 80% ให้สร้าง DAILY_COMPLETENESS_LOW ถ้าต้นทางส่ง status เป็น SHUTDOWN, Shutdown, หรือ Shut Down ให้ normalize เป็น Shut Down ก่อนคำนวณ ถ้า 24 - shutDownStatusCount = 0 ให้ถือว่าวันนั้นไม่มีชั่วโมงที่ต้องรายงาน และไม่สร้าง alert 80%
กติกาค่าผิดปกติ: ค่านิ่ง, ค่า 0, และค่าติดลบ ต้องเกิด 5 ครั้งติดกัน จึงสร้าง alert ถ้ามีค่าปกติกลับมา 1 ค่า ให้ reset sequence แล้วเริ่มนับใหม่ ในแต่ละวันให้เก็บช่วงแรกที่พบครบ 5 ครั้งติดกันไว้ใน alert_events โดย startedAt คือเวลาของค่าครั้งที่ 1 และ endedAt คือเวลาของค่าครั้งที่ 5 ที่ยืนยันว่าเข้าเงื่อนไขแล้ว

3. Flow งานที่แนะนำ

รูป Flow งานภาพรวม

1. รับข้อมูล 60m raw data หรือ POST จากทีมภายนอก
2. Detect job ตรวจเฉพาะช่วงใหม่ ไม่ scan ทั้งระบบ
3. Upsert สร้าง idempotency_key แล้วบันทึก alert_events
4. Notify worker ตรวจว่ายังไม่เคยส่ง แล้วส่งแจ้งเตือน
5. Read หน้าเว็บอ่านจาก alert_events เท่านั้น

Flow แบบข้อความ

External POST API (standard / EIA exceeded)
  -> Build idempotency_key
  -> Upsert alert_events

60m raw tables
  -> Hourly 60m Detector (ค่านิ่ง/ค่า 0/ค่าติดลบ 5 ครั้งติดกัน)
  -> Daily Summary Job
  -> 08:30 Alert Job (80% / ไม่รายงานต่อเนื่อง / ค่าผิดปกติ)
  -> Build idempotency_key
  -> Upsert alert_events

alert_events
  -> Notification Worker
  -> Insert alert_notifications
  -> Frontend reads alert_events with filters

4. Schema หลัก

4.1 alert_events

ตารางนี้เก็บเฉพาะรายการที่เข้าเงื่อนไขแจ้งเตือนแล้ว ไม่เก็บ raw 60m ทั้งหมดซ้ำ 

CREATE TABLE alert_events (
  id BIGINT IDENTITY(1,1) PRIMARY KEY,

  alert_type VARCHAR(64) NOT NULL,
  system_type VARCHAR(16) NOT NULL,
  display_system_type VARCHAR(32) NOT NULL,

  factory_id VARCHAR(64) NULL,
  factory_name NVARCHAR(500) NOT NULL,
  factory_registration_no NVARCHAR(64) NULL,

  connected_measurement_point_id BIGINT NULL,
  station_id VARCHAR(64) NOT NULL,
  point_name NVARCHAR(255) NOT NULL,
  point_code VARCHAR(64) NULL,
  point_type VARCHAR(32) NULL,

  parameter_code VARCHAR(64) NOT NULL,
  parameter_name NVARCHAR(255) NOT NULL,
  parameter_label NVARCHAR(255) NOT NULL,
  unit NVARCHAR(64) NULL,

  event_date DATE NOT NULL,
  started_at DATETIME2 NULL,
  ended_at DATETIME2 NULL,

  measured_value DECIMAL(18,6) NULL,
  threshold_value DECIMAL(18,6) NULL,
  threshold_type VARCHAR(32) NULL,

  completeness_percent DECIMAL(5,2) NULL,
  consecutive_days INT NULL,
  abnormal_type VARCHAR(32) NULL,

  source_table VARCHAR(128) NULL,
  source_interval VARCHAR(16) NOT NULL DEFAULT '60m',
  source_payload_json NVARCHAR(MAX) NULL,
  evidence_json NVARCHAR(MAX) NULL,

  notification_status VARCHAR(32) NOT NULL DEFAULT 'AUTO',

  idempotency_key VARCHAR(220) NOT NULL,
  detected_at DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
  created_at DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
  updated_at DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
  deleted_at DATETIME2 NULL
);

4.2 measurement_daily_summaries

ตารางสรุปรายวันช่วยให้เงื่อนไข 80%, ไม่รายงานต่อเนื่อง, ค่านิ่ง/0/ติดลบ อ่านเร็วขึ้นมาก 

CREATE TABLE measurement_daily_summaries (
  id BIGINT IDENTITY(1,1) PRIMARY KEY,

  system_type VARCHAR(16) NOT NULL,
  factory_id VARCHAR(64) NULL,
  station_id VARCHAR(64) NOT NULL,
  parameter_code VARCHAR(64) NOT NULL,
  parameter_label NVARCHAR(255) NOT NULL,
  unit NVARCHAR(64) NULL,

  summary_date DATE NOT NULL,
  expected_count INT NOT NULL,
  received_count INT NOT NULL,
  normal_status_count INT NOT NULL DEFAULT 0,
  shutdown_status_count INT NOT NULL DEFAULT 0,
  completeness_percent DECIMAL(5,2) NOT NULL,

  null_count INT NOT NULL DEFAULT 0,
  zero_count INT NOT NULL DEFAULT 0,
  negative_count INT NOT NULL DEFAULT 0,
  same_value_max_streak INT NOT NULL DEFAULT 0,

  first_received_at DATETIME2 NULL,
  last_received_at DATETIME2 NULL,
  source_table VARCHAR(128) NULL,

  created_at DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
  updated_at DATETIME2 NOT NULL DEFAULT SYSDATETIME()
);

5. Index และ Partition สำหรับข้อมูลใหญ่

กรณีมี 10,000 โรงงานต่อวัน และ 1 โรงงานมีหลายจุดตรวจ/หลายพารามิเตอร์ ต้องออกแบบให้ query ส่วนใหญ่ใช้ index ตามวันที่และประเภทแจ้งเตือน หลีกเลี่ยงการค้นแบบ full scan 
CREATE UNIQUE INDEX uq_alert_events_idempotency
ON alert_events(idempotency_key)
WHERE deleted_at IS NULL;

CREATE INDEX ix_alert_events_date_type
ON alert_events(event_date DESC, alert_type, system_type)
WHERE deleted_at IS NULL;

CREATE INDEX ix_alert_events_factory_date
ON alert_events(factory_id, event_date DESC)
WHERE deleted_at IS NULL;

CREATE INDEX ix_alert_events_station_param_date
ON alert_events(station_id, parameter_code, event_date DESC)
WHERE deleted_at IS NULL;

CREATE UNIQUE INDEX uq_daily_summary_station_param_date
ON measurement_daily_summaries(station_id, parameter_code, summary_date);

ถ้าปริมาณข้อมูลโตมาก ให้พิจารณา partition alert_events ตาม event_date รายเดือนหรือรายไตรมาส เพื่อให้ query รายเดือนและการ purge ข้อมูลเก่าทำได้เร็ว

6. API สำหรับทีมภายนอก POST ค่าแจ้งเตือน

API นี้ใช้รับเฉพาะเหตุการณ์ที่อีกทีมประเมินแล้วว่าเกินมาตรฐานหรือเกินค่า EIA เท่านั้น ฝั่ง POMS ไม่ต้องคำนวณซ้ำว่าเกินหรือไม่ แต่ต้อง upsert ด้วย idempotencyKey แล้วตรวจว่า event นี้เคยส่งแจ้งเตือนไปแล้วหรือยัง

Endpoint

POST /api/v1/integrations/alert-events
ขอบเขตของ API นี้: ระบบภายนอกต้อง POST มาเฉพาะรายการที่ยืนยันแล้วว่า เกินมาตรฐาน หรือ เกินค่า EIA เท่านั้น ฝั่ง POMS ทำหน้าที่ตรวจ idempotencyKey, บันทึก/อัปเดต alert_events, และเช็ค alert_notifications ว่าเคยส่งแจ้งเตือน event นี้แล้วหรือยัง

ข้อมูลที่ External API ต้องส่ง

ถ้าอีกทีมส่ง alert event มาให้ POMS โดยตรง ให้ส่งข้อมูลให้ครบพอสำหรับ 3 งาน: กันข้อมูลซ้ำ, แสดงผลในตาราง, และตรวจสอบย้อนหลังว่า event นี้อ้างอิงค่าจากไหน

กลุ่มข้อมูล Field Required ตัวอย่าง หมายเหตุ
กันข้อมูลซ้ำ idempotencyKey Yes CEMS:S0001:SO2:STANDARD_EXCEEDED:2026-03-02:20 ใช้ upsert กันยิงซ้ำหรือ job rerun แล้วเกิด event ซ้ำ
ประเภทระบบ systemType Yes CEMS, WPMS ในระบบเดิม BOD/COD Online ควร map เป็น WPMS
ชื่อแสดงผล displaySystemType Yes CEMS, BOD_COD_ONLINE ใช้ให้ frontend แสดงชื่อเมนู/รายงานชัดเจน
ประเภท alert alertType Yes STANDARD_EXCEEDED, EIA_EXCEEDED External API ส่งมาเฉพาะ alert ที่เกินมาตรฐาน/EIA แล้ว ส่วน 80%, no report, abnormal ให้ POMS คำนวณจาก 60m
โรงงาน factoryId Recommended factory-001 ถ้าไม่ส่ง POMS ต้อง lookup จาก stationId; ส่งมาจะช่วยลด ambiguity
โรงงาน factoryName Recommended บริษัท 2584 จำกัด ใช้เป็น snapshot แสดงผลย้อนหลัง แม้ชื่อใน master เปลี่ยนภายหลัง
โรงงาน factoryRegistrationNo Recommended 3-xx-xx ใช้แสดงคอลัมน์เลขทะเบียนโรงงานตามรูปตัวอย่าง
จุดตรวจวัด stationId Yes S0001, P0001 รหัสหลักสำหรับจับกับตาราง {stationId}_data_60m และจุดตรวจวัดใน POMS
จุดตรวจวัด pointCode Recommended S0001 ถ้า pointCode กับ stationId คือค่าเดียวกัน ส่งซ้ำได้เพื่อความชัดเจน
จุดตรวจวัด pointName Yes Stack 1, จุดที่ 1 ใช้แสดงคอลัมน์จุดตรวจวัด
จุดตรวจวัด pointType Optional STACK, WASTEWATER ใช้กรอง/จัดกลุ่มเพิ่มเติม
พารามิเตอร์ parameterCode Yes so2, nox, bod, cod ควรตรงกับ prefix ใน 60m เช่น so2_value, cod_value
พารามิเตอร์ parameterName Yes SO2, NOx, BOD, COD ชื่อสั้นของพารามิเตอร์
พารามิเตอร์ parameterLabel Yes SO2 (ppm), COD (mg/l) ต้องมีหน่วยใน label เพื่อแสดงผลไม่กำกวม
พารามิเตอร์ unit Yes ppm, mg/l ใช้แสดงคอลัมน์หน่วยและตรวจย้อนหลัง
เวลาเกิดเหตุ eventDate Yes 2026-03-02 วันที่ของข้อมูลตรวจวัด ไม่ใช่วันที่ส่ง API
เวลาเกิดเหตุ startedAt Yes 2026-03-02T20:00:00+07:00 เวลาเริ่มช่วงตรวจวัด เช่น 20.00
เวลาเกิดเหตุ endedAt Yes 2026-03-02T20:59:59+07:00 เวลาสิ้นสุดช่วงตรวจวัด เช่น 20.59
ค่าและเกณฑ์ measuredValue Yes 150 ค่าผลตรวจวัดที่เกิน ต้องเป็นตัวเลข
ค่าและเกณฑ์ thresholdValue Yes 60, 100 ค่าเกณฑ์ที่ใช้ตัดสินว่าเกิน
ค่าและเกณฑ์ thresholdType Yes STANDARD, EIA ต้องสัมพันธ์กับ alertType
สถานะแจ้งเตือน notificationStatus Optional AUTO, OFFICER ถ้าไม่ส่ง ให้ default เป็น AUTO
อ้างอิงต้นทาง sourcePayload Recommended { "externalReferenceId": "external-123" } เก็บ payload หรือ reference จากระบบภายนอกสำหรับตรวจสอบย้อนหลัง

Validation หลักที่ POMS ควรตรวจ

  • alertType ที่รับจาก external API ควรจำกัดที่ STANDARD_EXCEEDED และ EIA_EXCEEDED
  • systemType ต้องเป็น CEMS หรือ WPMS
  • thresholdType ต้องสัมพันธ์กับ alertType: STANDARD_EXCEEDED ใช้ STANDARD, EIA_EXCEEDED ใช้ EIA
  • measuredValue และ thresholdValue ต้องเป็นตัวเลข; ฝั่ง external เป็นผู้ยืนยันว่า event นี้เกินมาตรฐาน/EIA แล้ว
  • eventDate, startedAt, endedAt ต้องเป็นช่วงเวลาของข้อมูลตรวจวัด และ startedAt ต้องไม่มากกว่า endedAt
  • stationId และ parameterCode ควรตรวจว่ามีอยู่ในจุดตรวจวัดที่เชื่อมต่อแล้ว หรืออย่างน้อยบันทึกเป็น pending/error ถ้าไม่พบ master

8. กติกา idempotency_key

idempotency_key คือ key กันบันทึกซ้ำ เมื่อ job รันซ้ำหรือทีมภายนอก POST ซ้ำ ระบบต้องได้ event เดิม ไม่สร้างแถวใหม่

กรณี ตัวอย่าง key
เกินมาตรฐานรายชั่วโมง CEMS:S0001:SO2:STANDARD_EXCEEDED:2026-03-02:20
เกิน EIA รายชั่วโมง WPMS:P0001:COD:EIA_EXCEEDED:2026-03-02:20
รายงานไม่ถึง 80% ต่อวัน CEMS:S0001:NOX:DAILY_COMPLETENESS_LOW:2026-03-02
ไม่รายงานต่อเนื่อง WPMS:P0001:COD:CONSECUTIVE_NO_REPORT:2026-03-02:2026-03-09:ALERT_2026-03-10
ค่านิ่ง/0/ติดลบ CEMS:S0001:NOX:ABNORMAL_VALUE:CONSTANT:2026-03-02:FIRST

9. Job Strategy

Job รอบเวลา อ่านจาก เขียนไป
External exceeded event receiver เมื่อระบบภายนอก POST เข้ามา External API payload เฉพาะเกินมาตรฐาน/EIA alert_events และตรวจ alert_notifications ว่าเคยส่งแล้วหรือยัง
Hourly abnormal detector ทุกชั่วโมง หรือหลัง raw data เข้า 60m rolling window: ค่านิ่ง 5 ครั้งติดกัน, ค่า 0 5 ครั้งติดกัน, ค่าติดลบ 5 ครั้งติดกัน alert_events
Daily summary builder หลังจบวัน หรือ 08:00 60m ของวันก่อนหน้า measurement_daily_summaries
Daily completeness detector 08:30 measurement_daily_summaries alert_events
No report detector 08:30 measurement_daily_summaries หรือ last seen alert_events
Abnormal value detector ทุกชั่วโมง หรือ 08:30 60m rolling window / daily summary โดยใช้ threshold 5 ครั้งติดกัน alert_events

10. ขั้นตอนบน Windows Server

สำหรับ Windows Server ไม่ต้องใช้ Linux cron ให้ใช้ Windows Task Scheduler เรียก job script ตามเวลาที่กำหนด และให้ job ทุกตัวออกแบบเป็น idempotent คือรันซ้ำได้โดยไม่สร้างข้อมูลซ้ำ

10.4 ตั้ง Windows Task Scheduler

แนะนำให้สร้าง task แยกตามหน้าที่ เพื่อ debug ง่ายและรันซ้ำเฉพาะ job ได้

Task Trigger Action หมายเหตุ
Alert Daily Summary ทุกวัน 08:00 npm run job:alerts:summary สรุปข้อมูล 60m ของวันก่อนหน้า
Alert Daily Detection ทุกวัน 08:30 npm run job:alerts:daily ตรวจ 80%, ไม่รายงาน 7/14 วัน, และ event รายวันที่ต้องสร้าง
Alert Hourly Abnormal ทุกชั่วโมง npm run job:alerts:hourly ตรวจค่านิ่ง/ค่า 0/ค่าติดลบ 5 ครั้งติดกันจาก 60m
Alert Notification Worker ทุก 5 นาที หรือหลัง daily job npm run job:alerts:notify ส่งเฉพาะ event ที่ยังไม่มี log ส่งสำเร็จใน alert_notifications

10.6 Logging ที่ควรมี

  • log เริ่ม/จบ job พร้อมวันที่ที่ประมวลผล
  • จำนวน row ที่อ่านจาก 60m
  • จำนวน alert ที่สร้างใหม่
  • จำนวน alert ที่เจอซ้ำจาก idempotency_key
  • จำนวน notification ที่ส่งสำเร็จ/ล้มเหลว
  • error stack ของ job แต่ต้องไม่ log secret/API key

10.7 ขั้นตอนทดสอบก่อนขึ้น Production

  1. รัน migration บน staging DB
  2. ยิง POST /api/v1/integrations/alert-events 1 รายการ แล้วเช็คว่าเกิด alert_events
  3. ยิง POST ซ้ำด้วย idempotencyKey เดิม แล้วเช็คว่าไม่เกิด row ซ้ำ
  4. รัน npm run job:alerts:summary กับวันที่ทดสอบ
  5. รัน npm run job:alerts:daily แล้วเช็ค 80% และ no report
  6. seed 60m ให้มีค่านิ่ง/0/ติดลบ 5 ครั้งติดกัน แล้วรัน npm run job:alerts:hourly
  7. รัน notification worker แล้วเช็คว่า event เดิมไม่ส่งซ้ำ
  8. ทดสอบ GET /api/v1/alert-events พร้อม query ของทั้ง 10 ตาราง
  9. เปิด Task Scheduler history/log เพื่อยืนยันว่า task รันตามเวลา 08:00, 08:30 และรายชั่วโมง
จุดสำคัญบน Windows Server คืออย่าผูก job กับการเปิดหน้าเว็บ ให้ Task Scheduler เป็นตัวปลุกงาน และให้ทุก job ใช้ idempotency_key เพื่อให้รันซ้ำได้ปลอดภัยเมื่อ server retry หรือผู้ดูแลกด rerun เอง

11. ข้อแนะนำสุดท้าย

  1. อย่าแยก 10 ตารางตาม alert type เพราะ query, index, API, และ notification logic จะซ้ำกันมาก
  2. ใช้ alert_events เป็นตารางกลาง แล้วแยกเมนูด้วย filter หรือ view
  3. เก็บ raw 60m ไว้ที่เดิม ไม่ copy ทั้งหมดเข้าระบบแจ้งเตือน
  4. เพิ่ม measurement_daily_summaries เพื่อรองรับข้อมูลขนาดใหญ่และลดการ scan raw data
  5. ใช้ alert_notifications แยกจาก alert_events เพื่อรู้ว่าส่งแจ้งเตือนแล้วหรือยัง
  6. ทุก event ต้องมี idempotency_key เพื่อกัน job rerun หรือ external POST ซ้ำ
