เทียบ Flow ในภาพกับระบบจริง

ส่วนในภาพ ตาราง/โมดูลในระบบ สถานะ คำอธิบาย
ฐานข้อมูลโรงงาน กรมโรงงาน fac_import ผ่าน eligible-factory-candidates.repository.ts แหล่งต้นทางภายนอก ระบบอ่าน candidate จากฐานโรงงานภายนอก แล้วกรองรายการที่ยังไม่ถูกเลือกเข้า eligible_factories
ฐานข้อมูลโรงงาน ที่เข้าข่าย eligible_factories แยกจริง เก็บโรงงานที่คัดเลือกแล้ว พร้อมเลขทะเบียน ชื่อ ที่อยู่ จังหวัด พิกัด ข้อมูลการผลิต สถานะ และข้อมูลผู้เลือก
ฐานข้อมูลคำขอ ระบบคำขอเชื่อมต่อ cems_wpms_connection_requests, cems_wpms_measurement_points, device_connection_configs แยกจริง คำขอเก็บ snapshot ข้อมูลโรงงานในช่วงยื่นคำขอ จุดตรวจวัด และ config อุปกรณ์ที่ผูกกับคำขอด้วย request_id
ฐานข้อมูลโรงงาน/จุดตรวจวัดในระบบ POMS cems_wpms_connected_measurement_points, active device_connection_configs แยกจริง หลังคำขอเป็น CONNECTED ระบบ sync จุดตรวจวัดเป็น current/live และ active device config ใช้ request_id = NULL
เพิ่ม/แก้ไขข้อมูลจุดตรวจวัดโดยตรง factory_monitoring_point_forms, factory_monitoring_points แยกอีก workflow มีตาราง parent/child สำหรับบันทึกโรงงานและหลายจุดตรวจวัด แต่เอกสารปัจจุบันระบุว่ายังไม่เชื่อมกับตารางคำขอเชื่อมต่อหรือ current/live เดิม

ข้อสรุปการแยกข้อมูล

คำถาม คำตอบ
ข้อมูลโรงงานที่เข้าข่ายแยกจากฐานโรงงานทั่วไปไหม แยกอยู่ใน eligible_factories และมี unique ที่เลขทะเบียนโรงงานใหม่
คำขอเชื่อมต่อเก็บข้อมูลโรงงานแยกไหม แยกเป็น snapshot ใน cems_wpms_connection_requests เพื่อคงข้อมูลตามเวลาที่ยื่นคำขอ
จุดตรวจวัดที่ใช้งานจริงแยกจากคำขอไหม แยกเป็น cems_wpms_connected_measurement_points และชี้กลับด้วย source_request_id / source_measurement_point_id
ข้อมูลในเมนู dashboard ใช้ current/live หรือคำขอ dashboard โรงงานดึงจุดตรวจวัดจาก cems_wpms_connected_measurement_points ผ่าน repository สำหรับ connected points
มีช่องว่างจากภาพไหม มีเพียงจุดที่ต้องระวัง: ฟอร์ม factory_monitoring_point_forms เป็นชุดข้อมูลใหม่ที่ยังไม่ sync เข้า request/current ตามเอกสารล่าสุด

หลักฐานจากโค้ดและเอกสาร

  • backend/src/db/migrations/0017_create_eligible_factories.ts สร้างตาราง eligible_factories สำหรับโรงงานที่เข้าข่าย
  • backend/src/db/migrations/0019_create_cems_wpms_connection_requests.ts สร้างตารางคำขอและจุดตรวจวัดในคำขอ
  • backend/src/db/migrations/0021_extend_connection_request_forms.ts เพิ่ม request_id ใน device config เพื่อแยก snapshot ของคำขอ
  • backend/src/db/migrations/0030_separate_request_device_config_snapshots.ts สร้าง cems_wpms_connected_measurement_points และแยก active config ที่ request_id IS NULL
  • backend/src/modules/connection-requests/connection-requests.repository.ts มี syncConnectedMeasurementPoints เพื่อ copy จุดจากคำขอเป็น current/live
  • docs/monitoring-point-forms.md ระบุว่าฟอร์มข้อมูลจุดตรวจวัดแยกจาก workflow คำขอเชื่อมต่อเดิม

ข้อเสนอสำหรับการสื่อสารงาน: ให้เรียกข้อมูลในคำขอว่า “snapshot ตามวันที่ยื่นคำขอ” และเรียกข้อมูลใน POMS ว่า “current/live หลังเชื่อมต่อสำเร็จ” จะลดความสับสนระหว่างข้อมูลโรงงานเดียวกันที่ถูก copy ไปอยู่คนละชั้นของระบบ
