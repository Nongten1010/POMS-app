# Operator Normal Connection Point Code Workflow

Status: Implemented

## Trigger

คำขอเชื่อมต่อใหม่แบบปกติของผู้ประกอบการผ่านการอนุมัติแบบ และ backend ต้องออกรหัสให้จุดตรวจวัดที่ยังไม่มี `pointCode`.

## Goal

ให้ backend ออกรหัสจุดตรวจวัดของคำขอปกติแบบเรียงลำดับ แยกลำดับตามระบบและปี พ.ศ. โดย CEMS ใช้ `CEMS-NNNN/YYYY` และ WPMS ใช้ `WEMS-NNNN/YYYY`.

## Scope

- ใช้เฉพาะคำขอปกติของผู้ประกอบการ (`submissionSource` ไม่ใช่ `OFFICER_DIRECT_API`).
- CEMS ใช้ `CEMS-` ตามด้วยลำดับอย่างน้อย 4 หลัก `/` และปี พ.ศ. 4 หลัก เช่น `CEMS-0001/2569`.
- WPMS ใช้ `WEMS-` ตามด้วยลำดับอย่างน้อย 4 หลัก `/` และปี พ.ศ. 4 หลัก เช่น `WEMS-0001/2569`.
- CEMS และ WPMS มีลำดับแยกจากกัน และเริ่มที่ `0001` ใหม่เมื่อเปลี่ยนปี พ.ศ. ตาม `Asia/Bangkok`.
- หากคำขอมีหลายจุด ระบบจองเลขติดกันตามลำดับจุดในคำขอ.
- การจองเลขต้องปลอดภัยเมื่อมีหลายคำขอยืนยันพร้อมกัน.
- คำขอ `ADD_PARAMETER` ไม่ออกรหัสใหม่.
- การเชื่อมต่อโดยเจ้าหน้าที่โดยตรงยังใช้ `measurementPoints[].pointCode` ที่เจ้าหน้าที่กรอกเอง และไม่เข้าสู่ลำดับนี้.
- ไม่แก้ไขหรือแปลงรหัสเดิมที่มีอยู่ก่อน deploy รวมถึง `Sxxxx`, `Pxxxx` และ `Wxxxx`.
- หากมีรหัสรูปแบบใหม่ของระบบและปีเดียวกันอยู่แล้ว ให้ต่อจากเลขสูงสุดเพื่อหลีกเลี่ยงรหัสซ้ำ.
- รหัสเดิมและรหัสของปีอื่นไม่ถูกนำมาคำนวณลำดับของปีปัจจุบัน.

## Public Interface

ไม่มี endpoint ใหม่และ request payload ไม่เปลี่ยน พฤติกรรมที่สังเกตได้คือ เมื่อ status action ของคำขอปกติทำให้ backend ออกรหัสจุดตรวจวัด จุด CEMS แรกของปีได้ `CEMS-0001/YYYY` และจุด WPMS แรกของปีได้ `WEMS-0001/YYYY`; API รายละเอียดคำขอและ API จุดตรวจวัดที่เชื่อมต่อแล้วต้องคืนรหัสเดียวกัน. เมื่อใช้รหัสใน path parameter client ต้อง URL-encode `/` เป็น `%2F`; backend ต้องรองรับกรณี reverse proxy ถอด `%2F` เป็น `/` ก่อนส่งต่อด้วย.

## Failure And Consistency Rules

- การออกรหัสและการเปลี่ยนสถานะอยู่ใน transaction เดียวกัน.
- ห้ามออกรหัสซ้ำแม้เกิด concurrent requests.
- ถ้าการบันทึกสถานะหรือจุดตรวจวัดล้มเหลว ต้อง rollback การจองเลขตาม transaction behavior เดิม.

## TDD Behaviors

1. การอนุมัติคำขอ CEMS ปกติครั้งแรกในปี 2569 ออก `CEMS-0001/2569` และครั้งถัดไปออก `CEMS-0002/2569`.
2. การอนุมัติคำขอ WPMS ปกติลำดับที่ 3 ในปี 2571 ออก `WEMS-0003/2571`.
3. ลำดับ CEMS และ WPMS ไม่กระทบกัน.
4. เมื่อเปลี่ยนปี แต่ละระบบเริ่มลำดับใหม่ที่ `0001`.
5. Direct Connect ไม่ถูกสร้างรหัสอัตโนมัติและเก็บรหัสที่เจ้าหน้าที่ส่งมาเหมือนเดิม.
6. Route ที่รับ station ID ต้องประกอบ `CEMS-0001/2569` หรือ `WEMS-0003/2571` กลับได้เมื่อ reverse proxy ส่งมาเป็นสอง path segments.

## Resolved Decisions

- คงรหัสเดิมทุกค่าไว้โดยไม่ migrate; การเปลี่ยนแปลงมีผลเฉพาะรหัสที่ระบบออกใหม่หลัง deploy.
- ตัวนับเก็บแยกตาม `systemType + buddhistYear` และเริ่มที่ `0` เพื่อให้รหัสแรกของปีเป็น `0001`.
- หากข้อมูลเดิมมีรหัสรูปแบบใหม่ของปีเดียวกันที่สูงกว่า ระบบต้องออกเลขถัดจากค่าสูงสุด.
- การล้างข้อมูลก่อนขึ้นระบบจริงเป็นงาน deployment แยก ไม่ใช่หน้าที่ของ migration นี้.
